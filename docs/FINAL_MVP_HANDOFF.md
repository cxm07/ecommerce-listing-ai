## 2. 合并前必须解决的阻塞项

### C1：公共响应契约不一致

公共文档中的 `ApiResponse` 与后端 V1 的实际实现不一致：后端和前端使用结构化错误对象（如 `code/message/details`），而旧契约中 `error` 是字符串且 `data` 语义不同。

**处理要求：**先建立一个“契约对齐”PR，由双方确认后统一修改：

- `docs/API_CONTRACT.md` 中成功、业务失败、参数校验失败的响应样例；
- 前端 TypeScript 类型；
- 后端响应模型和异常处理；
- 对应测试。

不得只修改某一端来绕过此问题。公共 API、数据结构和状态机任何一方都不能单方面改变。

### C2：没有认证和资源授权

后端 V1 当前没有登录校验；知道任务 ID 的调用方理论上可以读取、修改、审批和导出别人的任务。它仅可作为**本地单用户演示**。

部署前必须实现：

- 服务端验证 JWT（建议 Supabase Auth 或同等方案）；
- 从已验证身份获得 `actor_id`，不能信任前端传入的身份；
- Task、Product、SKU、文件、导出的归属/访问校验；
- 至少覆盖“用户 B 不能访问用户 A 任务”的自动化测试；
- 前端接入登录态，并正确展示 401/403。

### I1：FastAPI 参数校验未统一封装

非法 UUID、缺少必填字段等情况目前会返回 FastAPI 默认 422（`detail`），不符合统一响应信封。

在 C1 的契约确定后，增加全局 `RequestValidationError` 处理器，并测试非法 UUID、空请求体和错误字段类型。

## 3. 后端下一步：严格按此顺序执行

### 阶段 A：先锁定契约

1. 以 PR #3 为前置，创建独立的契约对齐 PR；不要直接修改 PR #4 或 PR #5 来掩盖差异。
2. 在 PR 描述中明确记录 C1、C2、I1 的决定及兼容策略。
3. 契约 PR 合并后，让 PR #5 和 PR #4 重新基于已确认的基线测试；再按依赖顺序合并。

建议合并顺序：`PR #3 → 契约对齐 PR → 后端 PR #5（重基/修复后）→ 前端 PR #4（重基/验收后）`。如有冲突，以已确认的公共契约为准，不用 Mock 行为覆盖真实接口。

### 阶段 B：将后端从演示存储升级为真实持久化

保留现有 `WorkflowService` 和状态机边界，替换 `MemoryRepository` / `LocalFileStorage` 的生产实现：

- PostgreSQL / Supabase 数据库保存任务、文件、Product、SKU、Issue、文案、审批、审计日志；
- 按既有 `DATA_SCHEMA.md` 建立迁移、外键、索引、UUID、UTC 时间和金额精度；
- 源文件与导出文件放私有对象存储；下载前由服务端完成授权，使用签名 URL 或受控代理；
- 服务重启后，任务进度、审核记录和导出结果仍可恢复；
- 服务端密钥只放部署环境变量，绝不能进入前端、仓库或浏览器。

### 阶段 C：接入认证与多用户隔离

完成 C2 的 JWT 校验、资源归属和访问控制。若使用 Supabase，同时配置数据库 RLS；即使有 RLS，后端接口仍需做业务层授权判断。

### 阶段 D：补齐集成测试和前端异常体验

至少新增以下测试：

- 完整成功工作流；
- 非法 UUID / 无效请求体均返回统一错误信封；
- 未登录返回 401、越权返回 403；
- 修改 SKU/价格后由后端重新检测 Issue，前端不得直接“关闭 Issue”；
- 导出文件仅可被任务有权用户下载；
- 前端对 401、403、网络错误和异常响应信封显示可理解的提示。

### 阶段 E：部署与最终验收

1. 建立 staging：前端、后端、数据库、私有对象存储、认证环境变量齐全。
2. 增加 CI：前端测试/构建、后端测试、迁移检查；在 staging 做浏览器端到端测试。
3. 通过下方“最终 MVP 验收”后，再部署生产环境。

## 4. 最终 MVP 的验收标准

以下全部满足，才可以称为“完整可运行 MVP”：

- 新用户登录后可创建任务、上传真实 Excel，并查看解析结果；
- 修改 Product/SKU 后由后端复检，状态只能经 `WorkflowService` 合法流转；
- 商品审核、文案生成、文案审核和 Excel 导出完整可用；
- 后端重启后数据和导出记录仍存在；
- 用户无法读取、修改或下载其他用户资源；
- 所有成功、业务错误和参数错误均符合已确认的统一 API 契约；
- 前端、后端测试、构建、数据库迁移均通过；
- staging 浏览器端完成一次真实端到端流程并成功下载导出文件。

## 5. 不要做的事

- 不要为了快速合并而让前端回退为 Mock，或让前端伪造状态变化；
- 不要增加“前端直接关闭 Issue”的接口或逻辑，Issue 必须由后端检测结果驱动；
- 不要绕过状态机直接写任务状态；
- 不要把 Supabase service role、数据库密码、JWT 密钥放进前端或 Git；
- 不要在未解决 C1/C2 前宣称已具备生产部署条件。
