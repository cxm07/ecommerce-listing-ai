# 前端工作流设计说明

## 目标

在 `feature/frontend-workflow` 分支完成可演示的电商商品上新工作台前端。它覆盖任务列表、创建任务、Excel 上传、解析状态、商品与 SKU 审核、文案审核、导出结果以及问题/错误展示，并在后端接口逐步就绪前使用符合公共契约的 Mock 数据。

## 非目标

- 不修改 `docs/API_CONTRACT.md`、`docs/DATA_SCHEMA.md`、状态枚举或 `WorkflowService`。
- 不实现 Excel 解析、标准化、问题检测、模型调用、真实认证、真实存储或导出文件生成。
- 不新增 ERP、平台发布、RAG、Agent 或自动审批。
- 不在浏览器端存放任何高权限密钥。

## 体验与视觉方向

采用“流程工作台”作为任务详情的主骨架：左侧导航保持稳定，主区域根据任务状态展示当前阶段、下一步动作、禁用原因和相关问题。视觉采用已确认的低干扰业务工作台风格：克制的卡片边界、明确留白、状态胶囊、流程阶段条和可追溯审核时间线。

任务列表是入口；单个任务进入工作台后，始终能看见当前状态、待处理问题和下一步可执行动作。审核界面不把模型或系统结果伪装为人工确认，所有高风险动作必须显式进入人工审核路径。

## 页面与路由

| 路由 | 页面 | 主要责任 |
| --- | --- | --- |
| `/tasks` | 任务列表 | 展示任务状态、阻塞问题和下一步入口。 |
| `/tasks/new` | 创建任务 | 收集 `task_name` 与 `category`，创建 `DRAFT` 任务。 |
| `/tasks/:taskId/upload` | 上传 Excel | 选择受控 Excel，显示格式限制、上传进度与失败反馈。 |
| `/tasks/:taskId/processing` | 解析状态 | 展示 `UPLOADED`/`PARSING` 处理状态、来源文件和进入审核的结果。 |
| `/tasks/:taskId/products` | 商品审核 | 展示 Product、SKU、Issue 与 `source_ref`；允许在 `WAITING_PRODUCT_REVIEW` 修改已约定的产品/SKU字段并提交人工确认。 |
| `/tasks/:taskId/copy` | 文案审核 | 展示 `GeneratedContent`、`unsupported_claims` 和 `model_metadata`；仅允许人工批准进入 `APPROVED`。 |
| `/tasks/:taskId/export` | 导出结果 | 只在 `APPROVED`/`EXPORTED` 相关状态展示导出动作、结果文件和下载入口。 |
| `/tasks/:taskId/audit` | 审核记录 | 以时间线呈现审核动作、问题处理和来源定位。 |

现有 `/login` 保留为认证占位页；认证未接入时不把该页面当作权限实现。

## 前端模块边界

1. **页面模块**只负责组合视图、读取路由参数并触发用户动作。
2. **工作台模块**只依据公开 `TaskStatus` 决定阶段显示、可执行动作和禁用原因；它不自行推进状态。
3. **数据适配模块**提供一个小而稳定的读取/命令接口，先由 Mock 实现，后由 HTTP 实现替换。页面不能直接散落 Mock 常量或 `fetch` 调用。
4. **领域展示模型**严格映射公共字段：Task、TaskFile、Product、SKU、Issue、GeneratedContent、Approval 与 AuditLog。界面专用的派生文案（例如“下一步”）由状态映射生成，不能写回或假装成后端字段。
5. **共享 UI 模块**只承载展示与交互，不承载公共状态机或业务字段变换。

该设计使后端接入时只替换数据适配实现，而不要求重写路由或审核页面。

## Mock 与联调策略

- Mock 数据以 `sample-data/expected-result.json` 和 `sample-data/expected-issues.json` 的字段为基础，并补足公共数据结构中已有的标识、来源、审核和状态字段。
- 所有 Mock 命令返回与后端一致的 `{ status, data, issues, error }` 结构。
- Mock 状态推进只模拟公共状态机允许的动作；页面不会从 `WAITING_PRODUCT_REVIEW` 直接跳到 `APPROVED`。
- 后端提供接口后，将 HTTP 适配器接到相同模块接口；字段缺失或契约不一致时先提出独立公共契约 PR，不在前端私自兼容一套新字段。

## 状态反馈

每个页面都必须覆盖：

- 加载中：保留结构，告知正在读取或提交什么。
- 空状态：说明没有任务、没有问题或尚未有生成文案，并给出合法下一步。
- 错误状态：显示 API 的 `error` 与可定位 `issues`，不吞掉失败。
- 禁用状态：说明禁止原因，例如“商品事实未确认，不能生成文案”。
- 成功状态：明确显示动作已完成，但不把 `needs_review` 误标为完成。

## 测试接缝

测试通过用户可见接缝验证，而非测试私有实现细节：

1. 任务列表能显示状态、阻塞问题与正确下一步入口。
2. 创建任务表单在缺少必要字段时阻止提交，并在成功后创建 `DRAFT` 任务。
3. 上传页只接受 `.xlsx`，并在模拟上传失败时显示错误。
4. 商品审核页展示来源定位和问题；关键问题未处理时不能确认商品。
5. 文案审核页只在 `WAITING_COPY_REVIEW` 提供审核操作；审核前不能导出。
6. 导出页在非法状态展示禁用原因，在 `EXPORTED` 显示结果文件入口。
7. 审核记录按时间线展示人工动作与来源引用。

每个垂直切片使用“一个用户行为测试 → 最小实现 → 通过测试”的顺序；完整前端测试和构建在最终验收时运行。

## 协作约束

- 此分支只修改前端及前端测试；不修改后端处理逻辑。
- 公共 API、数据字段、状态机、统一响应格式、适配器接口有变化时停止当前实现，创建独立公共契约 PR 并与后端负责人确认。
- PR 需报告实际修改、测试结果、公共契约影响（预期为无）和遗留联调项。

## 风险与已知限制

- 当前后端除健康检查外主要提供契约，前端只能先以 Mock 完成可演示交互。
- 真实文件上传、解析进度、权限、持久化、模型元数据和下载文件需在后端实现后联调。
- 任务量大时的列表分页、搜索和权限过滤尚未被列入本 MVP 切片，不能在本轮自行扩大范围。
