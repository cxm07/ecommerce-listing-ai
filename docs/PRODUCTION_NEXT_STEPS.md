# 生产化交接：下一步

## 0. 边界与原则

当前交付物是**本地单用户可点击 MVP**，用于验证受控上新流程，不是生产系统。后续路由、前端、模型和 Agent 仍必须经 `WorkflowService` 推进状态。

**不得单方面变更公共 API 字段、响应信封、公共数据结构或任务状态机。**任何此类调整必须先更新 `docs/API_CONTRACT.md` / `docs/DATA_SCHEMA.md` / `docs/WORKFLOW.md`，并通过独立 Pull Request 评审；本交接文档不构成契约变更。

## 1. 已完成：本地 MVP

- FastAPI 已提供任务创建、上传、解析、商品/SKU 修正、审核、文案生成、导出、下载和工作台聚合读取的本地实现。
- React/Vite 工作台可通过 API 数据源运行；本地启动脚本会启动前后端并检查健康接口。
- 当前流程使用内存仓储与本地 `.local-data` 文件存储：服务重启后任务数据会丢失，不支持多用户隔离。
- 已有前端单测、前端生产构建和后端 pytest 的通过记录；完整 HTTP 链路曾验证至解析，后半段还需在无本机代理干扰的环境中回归。

## 2. 后端负责人：生产化任务

### 2.1 PostgreSQL 与迁移

1. 以现有 `DATA_SCHEMA.md` 为准建立迁移：`tasks`、`task_files`、`products`、`skus`、`issues`、`generated_content`、`approvals`、`skill_runs`、`audit_logs`，保留 UUID、UTC 时间、`numeric` 价格、JSONB 来源快照及外键索引。
2. 新增 PostgreSQL Repository，实现现有边界接口；在 feature flag/配置下替换 `MemoryRepository`，不得让路由直接访问数据库。
3. 在 staging 先执行空库迁移、回滚演练和样本数据导入验证；生产迁移须有备份、版本号、执行记录及回滚方案。
4. 迁移后回归上传、解析、修正 Issue、两次人工审核、导出和下载；状态推进只允许由 `WorkflowService` 执行。

### 2.2 Supabase Storage

1. 创建私有 source/export bucket；路径使用任务 ID、文件 ID 和不可冲突名称，原始上传文件只新增、不得覆盖。
2. `task_files.storage_path` 保存对象引用，下载由后端完成任务授权后生成短期签名 URL 或代理文件流。
3. 校验文件类型、大小、上传失败清理策略和导出文件的 Content-Type；记录上传/下载审计日志，禁止把 service-role key 交给浏览器。

### 2.3 认证与任务隔离

1. 接入 Supabase Auth 的服务端 JWT 校验，健康接口可保持匿名；其余业务接口从可信身份取得 `actor_id`，不接受客户端伪造的用户 ID。
2. 创建任务时写入 `creator_id`；每一次按任务、商品、SKU、文件访问都先校验所属用户/租户，再读取或变更数据与对象存储。
3. 数据库启用 RLS，并以服务端授权和 RLS 双重限制任务、关联记录及 Storage 对象；增加跨用户猜测 ID、越权下载和越权更新的集成测试。
4. 多租户字段、角色模型和共享协作规则尚未定稿；若需要，先提交独立契约/数据模型 PR，再实现。

### 2.4 验证错误统一信封

1. 盘点 Pydantic 请求校验、领域校验、未认证、未授权、未找到、冲突和内部错误的现状。
2. 提交独立 API 契约 PR，明确 HTTP 状态码与统一响应 `status/data/issues/error` 的字段含义；特别处理框架默认的 `422 {"detail": ...}`。
3. 契约获批后，以全局异常处理器和契约测试统一实现，不得在单个路由临时拼装不同错误格式。

### 2.5 部署环境

1. 分离 local、staging、production 配置；后端保存 Supabase URL、服务端密钥、JWT 配置、CORS 白名单、日志级别和 Storage bucket 名称，密钥仅放在部署平台的 secret 管理中。
2. 设置数据库连接池、健康/就绪检查、结构化日志、错误追踪、备份和告警；日志不得包含访问令牌、service-role key 或完整敏感上传内容。
3. 先部署 staging，完成迁移与端到端验收后再部署 production；前端只注入公开客户端配置和后端 API 地址。

## 3. 前端后续项

- 接入登录态并在 API 客户端发送访问令牌；401/403 要有明确提示，不能用本地 mock 掩盖授权失败。
- 继续以 `GET /api/tasks/{task_id}/workspace` 刷新聚合快照；修正 Product/SKU 后重新读取工作台，不新增“标记问题已解决”接口。
- 按服务端状态和 `issues` 驱动按钮禁用、加载、空状态与错误状态；前端不得自行推导新状态或绕过 `WorkflowService`。
- 对真实下载流、会话过期、网络失败和移动端完成端到端测试；下载仅调用约定的 `/download`，不使用 data URL mock。

## 4. 建议 PR 合并顺序

1. **契约基线 PR**：确认当前 API、数据结构、状态机和前后端兼容修正；只处理文档与兼容性差异。
2. **数据库迁移与 PostgreSQL Repository PR**：含迁移、仓储测试和 feature flag，不含认证语义变更。
3. **认证、RLS 与 Storage PR**：含任务隔离、私有文件访问和越权测试。
4. **错误信封契约 PR**：先评审 API 文档，再落地全局异常处理与前端适配。
5. **部署与前端认证/体验 PR**：staging 配置、监控、前端登录态与端到端回归。

每个 PR 均从最新集成基线创建，按依赖顺序合并；发生公共契约冲突时停止合并并回到独立契约 PR 解决。

- [ ] 原始上传和导出文件均在私有 Storage；跨用户访问、覆盖上传和未授权下载均被拒绝。
- [ ] 所有业务接口（健康检查除外）完成认证和任务所属校验；RLS 与后端授权测试通过。
- [ ] API 成功、验证失败、授权失败和领域失败均符合已批准的统一契约。
- [ ] 完整工作流在 staging 跑通：创建 → 上传 → 解析 → 修正 → 审核 → 生成 → 审核 → 导出 → 下载。
- [ ] 前端测试、生产构建、后端单测/集成测试、契约测试和端到端测试均通过。
- [ ] 密钥未进入仓库、前端构建产物或日志；CORS、环境变量、健康检查、监控、告警与备份已复核。
