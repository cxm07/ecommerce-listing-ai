# 扩展路线图

## 1. 使用原则

本路线图描述扩展的推荐顺序，不构成已承诺的交付时间表。每一阶段开始前，应先确认真实业务需求、样本、角色、合规约束和验收方式；未经验证的内容仍应保留为假设。

扩展必须遵守以下边界：

1. 不破坏当前公共数据结构、API 契约或 `WorkflowService` 状态机；若需要修改，先提交独立公共契约 PR。
2. 先建立可测试的确定性处理链路，再考虑模型、Agent 或外部系统自动化。
3. 新来源、新输出、模型、知识库、Agent 和第三方系统分别通过既有边界扩展，不将实现细节渗入路由或前端页面。
4. 每个阶段只扩展一个可验收的业务切片，并附带测试、日志与文档更新。

## 2. 阶段 0：公共基线

**目标**：让多人可以在同一套公共语义上并行开发。

- 补全产品范围、工作流、扩展路线图和设计决策文档；
- 固化当前数据结构、API 契约、状态机和适配器边界；
- 保持前端、后端、样本数据和 CI 骨架可运行。

**完成信号**：公共文档与代码中的状态枚举、接口命名和目录职责一致。

## 3. 阶段 1：第一条可验证纵向链路

**目标**：完成固定 Excel 的“创建任务 → 上传 → 保存来源文件 → 解析 → 返回标准化商品与 SKU”。

- 实现 `ExcelSourceAdapter`、固定模板校验和服务层；
- 保存来源文件及其不可覆盖的引用；
- 生成 Product、SKU、Issue，并保留来源行与原始字段快照；
- 通过 API 契约联调前端的任务创建、上传状态和商品审核骨架。

**不做**：真实 Supabase 接入、文案生成、导出、ERP 或平台集成。

## 4. 阶段 2：人工审核与导出闭环

**目标**：将问题处理、商品确认、文案审核和 Excel 导出串成完整受控流程。

- 完成商品/SKU 编辑、关键问题处理和审核记录；
- 接入可替换的 `ModelProvider`，仅基于确认后的事实生成文案；
- 保存生成元数据和不支持声明；
- 实现 `ExcelOutputAdapter` 和已审核结果导出；
- 增加工作流、异常分支与端到端样本测试。

**触发前提**：阶段 1 的字段映射、问题类型和人工审核动作已经由实际使用者验证。

## 5. 阶段 3：持久化、认证与可观测性

**目标**：用真实的存储、身份和审计能力替换演示骨架。

- 通过 Supabase PostgreSQL、Auth 和 Storage 实现持久化边界；
- 明确用户、任务访问和审核角色策略；
- 增加结构化日志、任务审计、失败诊断、超时和重试策略；
- 明确数据保留、删除和敏感信息处理要求。

**风险**：真实权限与数据保留要求必须来自业务调研，不能从 MVP 假设推导。

## 6. 阶段 4：受控集成与多模板

**目标**：在明确契约下扩展输入、输出与外部系统。

- 为新的文件格式或渠道实现独立 `SourceAdapter`、`OutputAdapter`；
- 通过 `IntegrationConnector` 对接经确认的 ERP 或平台能力；
- 添加模板版本识别、字段映射和回归样本；
- 对每个外部调用实施参数校验、幂等性、错误恢复和审计。

**前提**：真实系统所有者确认权限、字段、调用频率、回滚方式与责任边界。自动发布不应作为默认能力。

## 7. 阶段 5：知识与 Agent 能力（按需）

**目标**：只在确定性流程无法覆盖的场景引入可控智能能力。

- 为已验证的规范接入 `KnowledgeProvider`，保留引用与版本；
- 为可替换模型提供结构化输出、超时、重试和成本记录；
- 仅让 `AgentRuntime` 调用白名单 Skill，并把工具参数、结果和工作流跳转写入审计记录；
- 对不可信文档、提示注入和模型不确定性设置隔离与人工确认。

**不做**：为了“使用 Agent”而把确定性校验、状态推进或审批决策改为自由推理。

## 8. 变更检查清单

任何后续扩展开始前，确认：

- 是否改变了公共字段、接口字段或状态机？若是，是否已有独立契约 PR？
- 是否新增了外部数据、存储、模型或系统权限？是否明确授权范围和失败处理？
- 是否新增或更新了样本、测试、日志和用户可读的错误反馈？
- 是否把假设误写成了业务事实？若是，应移动到 [ASSUMPTIONS.md](ASSUMPTIONS.md)。
