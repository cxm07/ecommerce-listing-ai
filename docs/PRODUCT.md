# 产品说明

## 1. 产品定位

`ecommerce-listing-ai` 是一个用于验证电商商品资料整理与上新辅助流程的 Web 工作台 MVP。它帮助运营人员将一份受控格式的 Excel 商品资料，转换为可检查、可人工确认、可生成文案并可导出的上新结果。

本项目不是企业 ERP、商品发布平台或通用 Agent；当前也不替代既有业务系统。

## 2. MVP 要验证的问题

在固定 Excel 模板和单一演示类目的约束下，验证以下受控闭环是否清晰、可追溯：

```text
创建任务 → 上传 Excel → 解析商品和 SKU → 标准化与问题检查
→ 人工确认商品事实 → 生成文案 → 人工审核文案 → 导出 Excel
```

成功不等同于自动发布商品。成功是指：每一步都有明确输入、状态、人工责任和可回溯的来源信息。

## 3. 用户与职责

| 角色 | 在 MVP 中的职责 |
| --- | --- |
| 运营人员 | 创建任务、上传固定模板、修正或确认商品与 SKU 事实、审核文案、导出结果。 |
| 审核人 | 在需要时做商品事实或文案的人工审核；具体角色分工仍是 MVP 假设。 |
| 系统 | 保存任务状态、解析受控输入、提示问题、保留来源引用并生成待审核文案。 |

身份、权限模型和真实审批角色尚未实现，属于后续验证事项，详见 [ASSUMPTIONS.md](ASSUMPTIONS.md)。

## 4. 核心对象与结果

- **任务（Task）**：一次上新处理的工作单元，持有类别、状态和创建信息。
- **原始文件（TaskFile）**：不可覆盖的来源或导出文件记录。
- **商品与 SKU（Product / SKU）**：由来源文件解析并标准化的商品事实；来源行和原始字段快照必须可追溯。
- **问题（Issue）**：字段缺失、格式不一致或业务检查产生的结构化提示，需在确认前处理关键问题。
- **生成文案（GeneratedContent）**：基于已确认事实产生的标题和卖点；不应补全来源中不存在的商品事实。
- **审核记录（Approval）**：人工确认或拒绝的决策记录。

字段定义与约束以 [DATA_SCHEMA.md](DATA_SCHEMA.md) 为准；接口返回格式以 [API_CONTRACT.md](API_CONTRACT.md) 为准。

## 5. MVP 范围

### 包含

- 固定格式 Excel 的上传、解析、标准化与导出边界；
- 商品、SKU、问题、文案和审核的受控工作流；
- 人工确认商品事实与文案的两个闸门；
- 前端任务、创建任务、上传状态、商品审核和文案审核页面；
- 后端健康检查、统一响应模型、工作流状态机与扩展边界。

### 不包含

- ERP、CRM、店铺后台或平台发布 API；
- PDF、Word、图片、OCR 或多模板导入；
- RAG、真实 Agent、多 Agent 协作或自动决策审批；
- 真实生产认证、真实 Supabase 接入、多租户和生产级权限体系；
- 把未确认商品事实自动写入文案或发布目标。

## 6. 产品约束与验收原则

1. 系统只接受当前 MVP 约定的受控 Excel 输入；不匹配时应给出可理解的问题提示。
2. 原始文件、来源行与原始字段快照不可被后续操作覆盖。
3. 商品事实确认前不得生成文案；文案审核通过前不得导出。
4. 所有状态推进必须通过 `WorkflowService`，不得由页面、路由、模型或 Agent 绕过状态机。
5. 文案只陈述已经确认的商品事实，不生成无依据的认证、功效或绝对化承诺。
6. 当前规则仅用于 MVP 演示，未经业务调研不得视为正式企业制度。
